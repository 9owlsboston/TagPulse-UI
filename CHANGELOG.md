## Unreleased

### Fixed
- **Sprint 36 Phase E hotfix #4 — fourth blank-page outage caused by `antd-style` ↔ `antd-core` and `antd-form` ↔ `antd-core` ESM cycles, plus root-cause remediation.** A fourth TDZ crash hit production after PR #33: `antd-style-<hash>.js: Uncaught TypeError: i is not a constructor`. Root cause: `antd/es/style/*` (motion keyframes) does `new Keyframes("antFadeIn", {...})` at module-init, importing `Keyframes` from `antd-core`; antd-core in turn imports `antd/es/style/*` for default theme registration. The local fix (route `style` into `antd-core`) made the smoke pass — but then the new build immediately surfaced a *fifth* cycle: `antd-form-<hash>.js: ReferenceError: Cannot access 'it' before initialization` (the `g` export from antd-core, used at module-init by antd-form). Audit of `antd/es/_util/`, `antd/es/config-provider/`, `antd/es/locale/`, `antd/es/modal/`, `antd/es/theme/` (all in coreSet) showed they import from `button`, `form`, `tour`, `date-picker`, `pagination`, `popconfirm`, `table`, `upload`, `watermark`, and more — every per-component chunk forms a cycle with `antd-core`. Whichever binding antd-core's eager init touches first puts the cyclic importer's bindings in the TDZ window; the previous three hotfixes (#31, #32, #33) and this fourth one only addressed whichever cycle happened to trip up that build's evaluation order. **Robust fix:** [vite.config.ts](vite.config.ts) `manualChunks` now collapses every `antd/es/*` file into a single `antd-core` chunk so all cycles become intra-chunk (Rollup wraps them in one IIFE — no TDZ window possible). The per-component AntD split from Sprint 36 / #24 is retired (`react-vendor`, `recharts`, `leaflet`, `appinsights`, `@ant-design/icons` → `antd-core`, `@rc-component/*` → `antd-core`, and `rc-*` per-package splits remain). `antd-core` grows from 105 KB gz → 240 KB gz to absorb every AntD component, but the network cost in practice is small because most of those per-component chunks were already in antd-core's *eager* import graph (via `_util/ContextIsolator.js → form`, `locale → date-picker/table/upload/...`, `modal → button/watermark`, etc.), so the browser was already fetching them on first paint. Tree-shaking still works inside the single chunk: any `antd/es/<name>` we never import is dead-code-eliminated. [src/test/bundleChunkSeparation.test.ts](src/test/bundleChunkSeparation.test.ts) gains a fourth assertion pinning `antd/es/* → 'antd-core'` and asserting the legacy `return \`antd-${name}\`` fallback is gone, so a future per-component split can't quietly reintroduce the bug class.

### Added
- **Sprint 36 Phase E — `smoke:bundle` CI gate (Playwright + headless Chromium).** New [scripts/smoke-bundle.mjs](scripts/smoke-bundle.mjs) (~190 lines) boots the freshly built `dist/` in headless Chromium via Playwright, navigates to `/`, waits 5 s for the bundle to settle, and fails with exit 1 if **any** of: (a) a `pageerror` event fires, (b) a `console.error` includes a non-`Failed to load resource` message, (c) `/assets/*.js` fails to load, (d) `<div id="root">` has zero children (white-screen check). Exit 2 reserved for setup failures (server, browser launch). The script serves `dist/` from an in-process `node:http` static server with a path-traversal guard and an SPA index.html fallback that mirrors the Azure SWA `navigationFallback.exclude` rules — `vite preview` is unusable as the smoke target because its dev `server.proxy['/assets'] → :8000` collides with the built `build.assetsDir = 'assets'` and returns HTTP 500 for every chunk request. WSL2-safe chromium launch args (`--no-sandbox --disable-gpu --disable-software-rasterizer --disable-dev-shm-usage`) avoid the `FATAL:check.cc NOTREACHED` SIGTRAP after `Browser.setWindowBounds`. New npm scripts `smoke:bundle` and `build:smoke`. New devDep `playwright@^1.49.0`. This gate would have caught all four production hotfixes (PRs #31/#32/#33 + this PR's #4) before they shipped: validated end-to-end by running it against the broken bundles for each — it failed every one. CI wiring: a new `smoke` job in [.github/workflows/deploy-azure.yml](.github/workflows/deploy-azure.yml) caches `~/.cache/ms-playwright`, runs `npx playwright install chromium --with-deps`, `npm run build`, `npm run smoke:bundle`, and is a hard dependency of the deploy job — no future ESM-cycle bug can reach Azure SWA without an explicit override.

### Fixed
- **Sprint 36 Phase E hotfix #3 — third blank-page outage caused by `antd-icons` ↔ `antd-core` ESM cycle.** After PR #32 collapsed `@rc-component/*` into `antd-core`, production surfaced a third TDZ-family crash: `antd-icons-<hash>.js:58:1301 Uncaught TypeError: Cannot read properties of undefined (reading 'primary')`. Root cause: `@ant-design/icons` does `setTwoToneColor(blue.primary)` at module-init time, and the `blue` palette comes from `@ant-design/colors`, which lives inside `antd-core`. The cycle `antd-core → antd-icons → antd-core's blue palette` put the `blue` export in the TDZ window: when `antd-icons`'s IIFE ran, `antd-core`'s `oa.primary = oa[5]` initialiser hadn't fired yet, so the property access on `q.primary` (where `q = blue from antd-core`) threw on `undefined`. Same Azure-SWA-only manifestation as PRs #31/#32. Fix: [vite.config.ts](vite.config.ts) `manualChunks` now routes `@ant-design/icons` into `antd-core` instead of a separate `antd-icons` chunk. The palette binding is intra-chunk, Rollup wraps the initializer + the icons setup into the same IIFE, the TDZ window disappears. `antd-core` grows from 254 KB / 82 KB gz → 313 KB / 102 KB gz to absorb icons; the previous `antd-icons` chunk is gone. Tree-shaking of unused icons still works because each `@ant-design/icons/es/icons/<Name>` is its own ES module. [src/test/bundleChunkSeparation.test.ts](src/test/bundleChunkSeparation.test.ts) gets a third assertion pinning the `@ant-design/icons → 'antd-core'` rule. All 106 tests pass.
- **Sprint 36 Phase E hotfix #2 — second blank-page outage caused by `rc-component` ↔ `antd-core` ESM cycle.** After PR #31 broke the React/antd-core cycle, production surfaced a second TDZ crash from the same family of bug: `rc-component-<hash>.js:1 Uncaught ReferenceError: Cannot access 'un' before initialization`. Root cause: the `manualChunks(id)` rule collapsed every `@rc-component/*` package into a dedicated `rc-component` chunk, but `@rc-component/color-picker` re-exports the `FastColor` class through `antd/es/color-picker` (which the `core` Set kept inside `antd-core`), and uses it as a class superclass at module-init time: `gt = (function(r){...})(un)` where `un = F as un from "./antd-core-...js"`. AntD's `_util`, `select`, `dropdown`, `qr-code`, `tour`, and `watermark` also import from `@rc-component/*`, so `antd-core` and `rc-component` formed an ES-module cycle — when `rc-component`'s IIFE ran, `un`'s live binding hadn't been initialised by `antd-core` yet and the class superclass was in the TDZ. Same Azure-SWA-only manifestation as PR #31: `npm run dev`, vitest, and CI `check`+`preview` all stayed green because dev serves source ESM directly and CI never boots the bundle in a browser. Fix: [vite.config.ts](vite.config.ts) `manualChunks` now routes every `@rc-component/*` module into `antd-core` instead of a separate `rc-component` chunk. The cycle becomes intra-chunk, Rollup wraps the bindings into the same IIFE, and the TDZ window disappears. `antd-core` grows from 200 KB / 65 KB gz → 254 KB / 82 KB gz to absorb `@rc-component/{color-picker,trigger,qrcode,tour,mutate-observer,...}`; no other chunks change. [src/test/bundleChunkSeparation.test.ts](src/test/bundleChunkSeparation.test.ts) gets a second assertion pinning the `@rc-component/* → 'antd-core'` rule so a future refactor can't quietly re-introduce the cycle. All 104 tests still pass.
- **Sprint 36 Phase E hotfix — blank-page outage on every route after the AntD chunk split (Sprint 36 / #24 regression).** The `manualChunks(id)` function added in Sprint 36 / #24 pinned every AntD subcomponent + every `rc-*` package to its own chunk, but left `react`, `react-dom`, `scheduler`, and `react-is` un-pinned. Rollup's automatic chunk placement then hoisted React's production runtime into `antd-core` (the most-shared importer of `react`), which created an ES-module evaluation cycle: `antd-core` imports `antd-button` / `antd-modal` / `antd-message` / `antd-notification` / `antd-icons` / `antd-date-picker`, and those modules re-import React from `antd-core`. At runtime React's `q.Activity = …` exports initialiser ran with `q` (its exports object) still `undefined`, crashing the entire app with `Uncaught TypeError: Cannot set properties of undefined (setting 'Activity')` on **every** page (Dashboard, AssetDetail, /admin/labels, … all blank). Local `npm run dev` was unaffected because Vite serves source ESM directly (no Rollup chunking); the bug only manifested in the Azure SWA deployed bundle. Fix: [vite.config.ts](vite.config.ts) `manualChunks` now pins `react` / `react-dom` / `scheduler` / `react-is` to a dedicated `react-vendor` chunk (199.6 KB / 61.9 KB gz), anchored with trailing slashes so it doesn't accidentally swallow `react-router-dom` / `react-grid-layout` / `react-leaflet`. With React isolated, the cycle is broken. New entry chunk drops from 308 KB / 96 KB gz → 126 KB / 39 KB gz (the React payload is now shared via `react-vendor` instead of being part of the entry). New [src/test/bundleChunkSeparation.test.ts](src/test/bundleChunkSeparation.test.ts) enforces the `react-vendor` rule statically so a future `manualChunks` edit can't quietly regress the contract. All 104 tests pass.

### Changed
- **Sprint 36 / #24 — Modularize AntD imports to shrink the eager vendor chunk.** All 52 source files that did `import { Button, Form, … } from 'antd'` were converted (via [scripts/convert-antd-imports.py](scripts/convert-antd-imports.py), checked into the repo for reproducibility) to per-component direct imports `import Button from 'antd/es/button'`. This eliminates the barrel re-export file (`node_modules/antd/es/index.js`) as a static dependency, which previously forced Rollup to drag every AntD submodule into the eager graph: `Dashboard.tsx` doing `import { List } from 'antd'` pulled in `Table`, `DatePicker`, `Select`, `Tree`, `Modal`, `Upload`, `Form`, etc. via the barrel's re-export chain. [vite.config.ts](vite.config.ts) replaces the previous `manualChunks: { antd: ['antd'] }` (which forced the entire AntD library into one 378 KB gz chunk that was always preloaded) with a function form that gives each meaningfully sized AntD component (`Table`, `Form`, `Modal`, `Drawer`, `DatePicker`, `Select`, `Tree`, `Upload`, `Tabs`, `Menu`, `Input`, `Result`, `Typography`, …) its own chunk, and similarly splits every `rc-*` internal package. Tiny common primitives (`Tag`, `Space`, `Grid`, `Row`, `Col`, `Empty`, `Avatar`, `Switch`, …) collapse into a single `antd-core` chunk alongside the shared `_util` infrastructure. **Before:** single `antd` chunk = **377.66 KB gz** (1.21 MB raw), page-1 cold-load (entry + antd + antd-icons + Dashboard) = **~506 KB gz**. **After:** `antd-core` = **56.0 KB gz** (acceptance criterion ≤250 KB gz comfortably met), page-1 cold-load = **~434 KB gz** (≈73 KB gz / 14 % reduction), and `npm run build` no longer trips Vite's `chunkSizeWarningLimit` on the antd chunk. Heavy components (Table, DatePicker, Tree, Upload, Image, Tour, Cascader, Mentions, …) now ride lazily with the per-route page chunks that actually use them, so a deploy that touches only the `<Table>` source path invalidates only `antd-table` + `rc-table` for downstream caches instead of the full 378 KB blob. All 94 vitest tests still pass; no visual regressions on Dashboard, DeviceList, RuleEditor, CategoryList, or Branding (verified via `npm run dev` + manual smoke).
